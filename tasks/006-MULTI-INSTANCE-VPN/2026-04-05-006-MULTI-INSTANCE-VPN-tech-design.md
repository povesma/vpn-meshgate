# 006-MULTI-INSTANCE-VPN: Multi-Instance VPN Routing - Technical Design

**Status**: Draft
**PRD**: [006-MULTI-INSTANCE-VPN-prd.md](2026-04-05-006-MULTI-INSTANCE-VPN-prd.md)
**Created**: 2026-04-05

## Overview

Replace the single hardcoded L2TP company VPN with N dynamically
generated VPN instances, each with its own tunnel type
(L2TP/IPsec, WireGuard, OpenVPN), credentials, and CIDR routing.
A Python generator reads `vpn-instances.yaml` and produces
`docker-compose.override.yml`. Separate Dockerfile + entrypoint
per tunnel type. Route-init reads the same YAML to build
per-instance routing tables.

## Current Architecture

```
gluetun namespace (shared: tailscale, route-init, ntfy, healthcheck, vpn-bot)
  │
  ├─ COMPANY_CIDRS + EXTRA_VPN_CIDRS ──→ 172.29.0.20 (l2tp-vpn) ──→ ppp0
  └─ default ──→ tun0 (Mullvad WireGuard)
```

**Single-instance assumptions baked into**:
- `docker-compose.yml` — `l2tp-vpn` service hardcoded at
  `172.29.0.20` with L2TP-specific env vars
- `gluetun/init-routes.sh` — all CIDRs route to one IP
  (`172.29.0.20`)
- `l2tp/entrypoint.sh` — reads `COMPANY_CIDRS` +
  `EXTRA_VPN_CIDRS`, routes all over single `ppp0`
- `bot/bot.sh` — `restart company` / `disable company` assume
  single `l2tp-vpn` container
- `healthcheck/check.sh` — single `L2TP_CHECK_IP`
- `.env` — flat `L2TP_*` vars, single `COMPANY_CIDRS`

## Proposed Design

### Architecture (Target State)

```
gluetun namespace
  │
  ├─ instance "company-a" CIDRs ──→ 172.29.0.101 (vpn-company-a, l2tp)
  ├─ instance "company-b" CIDRs ──→ 172.29.0.102 (vpn-company-b, wireguard)
  ├─ instance "company-c" CIDRs ──→ 172.29.0.103 (vpn-company-c, openvpn)
  └─ default ──→ tun0 (Mullvad WireGuard)
```

VPN instance containers live on `bridge_vpn` starting at
`172.29.0.101` (incrementing per instance). The `.20` and `.30`
addresses remain reserved for backward-compatible dnsmasq.

### Component Boundaries

```
vpn-instances.yaml         (user-authored config, gitignored)
vpn-instances.yaml.example (committed, documented)
        │
        ▼
generate-vpn.py            (build-time: YAML → compose override)
        │
        ├──→ docker-compose.override.yml  (generated, gitignored)
        └──→ /shared/vpn-instances.json   (runtime: consumed by
                                            route-init, bot,
                                            healthcheck)
        ▼
docker compose up -d --build
        │
        ├──→ vpn-{name} containers (one per instance)
        ├──→ route-init (reads /shared/vpn-instances.json)
        ├──→ healthcheck (reads /shared/vpn-instances.json)
        └──→ vpn-bot (reads /shared/vpn-instances.json)
```

### Data Contract: `vpn-instances.yaml`

```yaml
instances:
  - name: company-a
    type: l2tp
    server: vpn.company-a.com
    cidrs:
      - 10.11.0.0/16
      - 10.12.0.0/16
    dns_domains:
      - company-a.com
    check_ip: 10.11.0.1
    credentials:
      username: user
      password: pass
      psk: shared-secret

  - name: company-b
    type: wireguard
    cidrs:
      - 10.21.0.0/16
    check_ip: 10.21.0.1
    config_file: wireguard/compb.conf

  - name: company-c
    type: openvpn
    cidrs:
      - 10.31.0.0/16
    check_ip: 10.31.0.1
    config_file: openvpn/compc.ovpn
    credentials:
      username: user
      password: pass

  - name: company-d
    type: netbird
    cidrs:
      - 10.41.0.0/16
    check_ip: 10.41.0.1
    credentials:
      setup_key: XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX
    # optional: management_url: https://api.netbird.io:443
```

**Validation rules** (enforced by `generate-vpn.py`):
- `name`: unique, alphanumeric + hyphens, used as container
  suffix (`vpn-{name}`)
- `type`: one of `l2tp`, `wireguard`, `openvpn`, `netbird`
- `cidrs`: non-empty, no overlaps across instances
- `credentials`: required fields vary by type (l2tp: username,
  password, psk; openvpn: username, password; netbird:
  setup_key; wireguard: none — uses config_file)
- `config_file`: required for wireguard and openvpn types
- `management_url`: optional for netbird (defaults to Netbird
  cloud)
- `check_ip`: optional, used for health monitoring
- `dns_domains`: optional, used by dnsmasq for split DNS

### Data Contract: `/shared/vpn-instances.json`

Generated by `generate-vpn.py`, mounted into shared-config
volume. Consumed at runtime by route-init, healthcheck, bot.

```json
[
  {
    "name": "company-a",
    "type": "l2tp",
    "ip": "172.29.0.101",
    "cidrs": ["10.11.0.0/16", "10.12.0.0/16"],
    "check_ip": "10.11.0.1",
    "dns_domains": ["company-a.com"],
    "container": "vpn-company-a"
  },
  {
    "name": "company-b",
    "type": "wireguard",
    "ip": "172.29.0.102",
    "cidrs": ["10.21.0.0/16"],
    "check_ip": "10.21.0.1",
    "dns_domains": [],
    "container": "vpn-company-b"
  }
]
```

### Component: `generate-vpn.py`

**Responsibility**: Read `vpn-instances.yaml`, validate, produce
`docker-compose.override.yml` and `/shared/vpn-instances.json`.

**IP assignment**: Instances get `172.29.0.101`, `.102`, `.103`
etc. in order of definition. Deterministic — same YAML always
produces same IPs.

**Generated compose override** per instance:

For L2TP (`type: l2tp`):
```yaml
services:
  vpn-company-a:
    build: ./l2tp
    container_name: vpn-company-a
    privileged: true
    dns: [1.1.1.1, 8.8.8.8]
    devices: ["/dev/ppp:/dev/ppp"]
    environment:
      - VPN_INSTANCE_NAME=company-a
      - L2TP_SERVER=vpn.company-a.com
      - L2TP_USERNAME=user
      - L2TP_PASSWORD=pass
      - L2TP_PSK=shared-secret
      - INSTANCE_CIDRS=10.11.0.0/16,10.12.0.0/16
      - NTFY_TOPIC=${NTFY_TOPIC:-vpn-alerts}
      - L2TP_CHECK_IP=10.11.0.1
    networks:
      bridge_vpn:
        ipv4_address: 172.29.0.101
    volumes:
      - shared-config:/shared
    restart: unless-stopped
```

For WireGuard (`type: wireguard`):
```yaml
services:
  vpn-company-b:
    build: ./wireguard
    container_name: vpn-company-b
    cap_add: [NET_ADMIN]
    environment:
      - VPN_INSTANCE_NAME=company-b
      - INSTANCE_CIDRS=10.21.0.0/16
      - NTFY_TOPIC=${NTFY_TOPIC:-vpn-alerts}
      - WG_CHECK_IP=10.21.0.1
    networks:
      bridge_vpn:
        ipv4_address: 172.29.0.102
    volumes:
      - shared-config:/shared
      - ./wireguard/compb.conf:/etc/wireguard/wg0.conf:ro
    restart: unless-stopped
```

For OpenVPN (`type: openvpn`):
```yaml
services:
  vpn-company-c:
    build: ./openvpn
    container_name: vpn-company-c
    cap_add: [NET_ADMIN]
    devices: ["/dev/net/tun:/dev/net/tun"]
    environment:
      - VPN_INSTANCE_NAME=company-c
      - INSTANCE_CIDRS=10.31.0.0/16
      - OVPN_USERNAME=user
      - OVPN_PASSWORD=pass
      - NTFY_TOPIC=${NTFY_TOPIC:-vpn-alerts}
      - OVPN_CHECK_IP=10.31.0.1
    networks:
      bridge_vpn:
        ipv4_address: 172.29.0.103
    volumes:
      - shared-config:/shared
      - ./openvpn/compc.ovpn:/etc/openvpn/client.conf:ro
    restart: unless-stopped
```

For Netbird (`type: netbird`):
```yaml
services:
  vpn-company-d:
    image: netbirdio/netbird:latest
    container_name: vpn-company-d
    cap_add: [NET_ADMIN, SYS_ADMIN, SYS_RESOURCE]
    environment:
      - NB_SETUP_KEY=XXXXXXXX-...
      - NB_MANAGEMENT_URL=https://api.netbird.io:443
      - VPN_INSTANCE_NAME=company-d
      - INSTANCE_CIDRS=10.41.0.0/16
      - NTFY_TOPIC=${NTFY_TOPIC:-vpn-alerts}
      - NB_CHECK_IP=10.41.0.1
    networks:
      bridge_vpn:
        ipv4_address: 172.29.0.104
    volumes:
      - shared-config:/shared
      - netbird-company-d:/var/lib/netbird
      - ./netbird/entrypoint.sh:/entrypoint.sh:ro
    entrypoint: ["/entrypoint.sh"]
    restart: unless-stopped
```

Note: Netbird uses the official `netbirdio/netbird` image (not
a custom build). A wrapper entrypoint is mounted to handle CIDR
routing, MASQUERADE, and monitoring — the same contract as other
types. A named volume persists Netbird state across restarts.

**The generator also removes the old `l2tp-vpn` service** from
the base `docker-compose.yml` (or the override sets
`profiles: [disabled]` to suppress it).

### Component: Per-Type Entrypoints

Each type has its own directory with Dockerfile + entrypoint.
All entrypoints follow the same contract:

1. Read `INSTANCE_CIDRS` (comma-separated)
2. Establish tunnel (type-specific)
3. Route `INSTANCE_CIDRS` over tunnel interface
4. Add MASQUERADE + MSS clamping
5. Monitor tunnel, notify via ntfy on disconnect
6. Auto-reconnect with backoff

**`l2tp/entrypoint.sh`** — Refactor existing script:
- Replace `COMPANY_CIDRS` / `EXTRA_VPN_CIDRS` with
  `INSTANCE_CIDRS`
- Replace hardcoded `[l2tp]` log prefix with
  `[vpn-${VPN_INSTANCE_NAME}]`
- Keep all existing logic (IPsec, xl2tpd, ppp0, reconnect loop)

**`wireguard/entrypoint.sh`** — New:
- `wg-quick up wg0` (config mounted at `/etc/wireguard/wg0.conf`)
- Route `INSTANCE_CIDRS` over `wg0`
- MASQUERADE + MSS clamping on `wg0`
- Monitor: `wg show wg0` for handshake freshness
- Reconnect: `wg-quick down wg0 && wg-quick up wg0`

**`openvpn/entrypoint.sh`** — New:
- Write auth file from `OVPN_USERNAME` / `OVPN_PASSWORD`
- `openvpn --config /etc/openvpn/client.conf --auth-user-pass`
- Route `INSTANCE_CIDRS` over `tun0`
- MASQUERADE + MSS clamping on `tun0`
- Monitor: check `tun0` interface exists
- Reconnect: restart openvpn process

**`netbird/entrypoint.sh`** — New:
- Runs the official `/usr/local/bin/netbird-entrypoint.sh` in
  background (starts full daemon: `netbird service run` +
  `netbird up`). This is required for DNS resolver, network
  routes, and domain-based routing to work. `--foreground-mode`
  disables these features and must NOT be used.
- Wait for `wt0` interface to appear
- Route `INSTANCE_CIDRS` over `wt0`
- MASQUERADE + MSS clamping on `wt0`
- DNS proxy: DNAT port 53 from bridge IP (eth0) to Netbird
  overlay IP (wt0), so dnsmasq can forward domain queries to
  Netbird's DNS resolver. Writes bridge IP to
  `/shared/{name}-dns-ip` for dnsmasq to discover.
- Monitor: check `wt0` interface exists
- On tunnel loss: kill daemon, exit. Docker restart policy
  restarts the container.
- Uses official `netbirdio/netbird` image; entrypoint mounted
  as override

### Netbird Domain-Based Routing

Netbird uses domain-based routing, not CIDR-based. The
management server pushes domain routes (e.g. `git.company.com`)
to peers. When Netbird's DNS resolver handles a query for a
routed domain, it resolves the IP AND automatically adds it to
WireGuard's allowed-IPs for the gateway peer. This means:

- `cidrs` in vpn-instances.yaml should contain only the Netbird
  overlay network (e.g. `10.75.0.0/16`), NOT the public IPs of
  company services
- `dns_domains` must list the company domain(s) so dnsmasq
  forwards queries to the Netbird DNS resolver
- Public IPs are routed dynamically — no need to list them in
  `VPN_ADVERTISE_ROUTES` or Headscale

**DNS resolution path for Netbird domains:**
```
Mac → Tailscale → dnsmasq (172.29.0.30)
  → Netbird DNS (172.29.0.102:53, DNAT to 100.75.x.x:53)
  → resolves domain + adds WireGuard route to gateway peer
  → traffic flows: wt0 → gateway peer → destination
```

### Component: `gluetun/init-routes.sh` (Modified)

**Current**: Reads `COMPANY_CIDRS` + `EXTRA_VPN_CIDRS` env vars,
routes all to single IP `172.29.0.20`.

**New**: Reads `/shared/vpn-instances.json`, routes each
instance's CIDRs to that instance's IP.

```sh
# Parse JSON with jq (add to route-init Dockerfile)
for row in $(jq -c '.[]' /shared/vpn-instances.json); do
    name=$(echo "$row" | jq -r '.name')
    ip=$(echo "$row" | jq -r '.ip')
    cidrs=$(echo "$row" | jq -r '.cidrs[]')
    for cidr in $cidrs; do
        ip route replace "$cidr" via "$ip" dev eth0
        ip rule add to "$cidr" lookup main priority 100
        iptables -A OUTPUT -o eth0 -d "$cidr" -j ACCEPT
        iptables -A INPUT -i eth0 -s "$cidr" -j ACCEPT
    done
done
```

`EXTRA_VPN_CIDRS` support is preserved — any CIDRs not assigned
to a specific instance route to the first instance (or a
configurable default).

### Component: `bot/bot.sh` (Modified)

Replace hardcoded `l2tp-vpn` references with instance-aware
commands:

- `status` — iterate `/shared/vpn-instances.json`, ping each
  `check_ip`, report per-instance
- `restart <name>` — `docker restart vpn-<name>`
- `disable <name>` — `docker stop vpn-<name> && docker update
  --restart=no vpn-<name>`
- `restart company` — backward compat: restart all VPN instances
- Keep `help` updated with new commands

### Component: `healthcheck/check.sh` (Modified)

Read `/shared/vpn-instances.json`. For each instance with a
`check_ip`, ping it. Report per-instance failures with instance
name in ntfy alert title.

### Component: `dns/entrypoint.sh` (Modified)

Read `/shared/vpn-instances.json` for `dns_domains` per
instance. Each instance's domains resolve via that instance's
DNS server (obtained from tunnel, written to
`/shared/{name}-dns-ip`).

### Migration from Current Config

**Removed from `.env`**: `L2TP_SERVER`, `L2TP_USERNAME`,
`L2TP_PASSWORD`, `L2TP_PSK`, `COMPANY_CIDRS`, `EXTRA_VPN_CIDRS`,
`COMPANY_DOMAIN`, `L2TP_CHECK_IP`. All moved to
`secrets/vpn-instances.yaml`.

**Added to `.env`**:
- `VPN_DNS_DOMAINS` — all `dns_domains` from all instances in
  `*.domain` format. Used by gluetun for DNS rebinding protection.
- `VPN_ADVERTISE_ROUTES` — all CIDRs from all instances. Used by
  Tailscale `--advertise-routes`. Required because Tailscale exit
  node drops private-subnet traffic unless explicitly advertised.
  Routes must be approved in Headscale.

**Stays in `.env`**: Mullvad (`WIREGUARD_*`, `MULLVAD_COUNTRY`),
Tailscale (`TS_*`, `HEADSCALE_URL`), ntfy, `VPS_PUBLIC_IP`,
`GLUETUN_API_KEY`, deployment vars.

### Secrets Architecture

Credentials never appear in compose files. The generator writes
per-instance `.env.vpn-{name}` files (0600 permissions) and the
compose override references them via `env_file:` directives.

**Secret storage locations**:
- `.env` — Mullvad, Tailscale, gluetun API credentials
- `secrets/vpn-instances.yaml` — VPN instance config with
  credentials (local-only, never synced to VPS)
- `.env.vpn-*` — per-instance credentials generated by
  `generate-vpn.py` (synced to VPS)

All three are gitignored. `secrets/` directory is fully gitignored
except `*.example` files. See CLAUDE.md for the full secrets
policy.

### Deployment Workflow

```
1. Edit secrets/vpn-instances.yaml
2. python3 generate-vpn.py
   → validates YAML
   → writes docker-compose.override.yml (no secrets)
   → writes vpn-instances.json (no secrets)
   → writes .env.vpn-* per instance (secrets, 0600)
3. Update .env if CIDRs/domains changed:
   VPN_DNS_DOMAINS and VPN_ADVERTISE_ROUTES
4. ./deploy-push.sh --force
   → syncs .env, .env.vpn-*, override, json, code
5. ./rdocker.sh compose up -d --build
6. If CIDRs changed: re-approve routes in Headscale:
   headscale nodes approve-routes \
     --identifier <ID> \
     --routes 0.0.0.0/0,::/0,<all-cidrs>
```

`generate-vpn.py` runs locally only.
`secrets/vpn-instances.yaml` never leaves your machine.

### Key Implementation Lessons

**Tailscale exit node drops private CIDRs**: Even in exit node
mode, Tailscale silently drops traffic to private subnets
(10.x, 172.x, 192.168.x) unless they are explicitly listed in
`--advertise-routes` and approved in Headscale. This was not
obvious — traffic appeared to route correctly inside the gluetun
namespace but never arrived from the Tailscale tunnel.

**Gluetun firewall parser breaks on fwmark**: The iptables rule
`-m mark --mark 0x80000/0xff0000` cannot be in `post-rules.txt`
because gluetun's firewall parser cannot handle the mark format
and enters a restart loop. This rule is now applied by route-init
instead.

**Dnsmasq must not depend on gluetun health**: Gluetun uses
dnsmasq (172.29.0.30) as its upstream DNS resolver. If dnsmasq
depends on `gluetun: service_healthy`, a circular dependency
forms — gluetun can't become healthy without DNS, dnsmasq won't
start without healthy gluetun.

**vpn-instances.json must be bind-mounted**: The `shared-config`
Docker volume is a named volume, not a bind mount. Files synced
via rsync to the VPS project directory don't appear inside it.
`vpn-instances.json` is bind-mounted directly into containers
that need it.

## Trade-offs

### Dynamic compose generation vs manual services

**Rejected: Manual services**. For 2 instances, manual is
feasible. For N, it becomes error-prone and violates DRY. The
generator adds a build step but ensures consistency between
routing config and container definitions.

### Universal container vs per-type containers

**Rejected: Universal container**. A single image containing
strongSwan + xl2tpd + wireguard-tools + openvpn would be large
and harder to debug. Per-type images are minimal, focused, and
follow the existing `l2tp/` pattern. Adding a new type means
adding a new directory, not modifying a shared entrypoint.

### Python generator vs shell + yq

**Chose Python**. YAML parsing, JSON generation, and validation
logic are far cleaner in Python than shell + yq. The generator
is a build-time tool, not a container runtime dependency.
PyYAML is the only dependency.

### JSON runtime config vs env vars

**Chose JSON** (`/shared/vpn-instances.json`). Route-init, bot,
and healthcheck all need the full instance list at runtime.
Passing this as env vars would require complex encoding. A JSON
file on the shared volume is readable by `jq` in shell scripts.

## Dependencies

**Build-time** (local machine):
- Python 3 + PyYAML (`pip install pyyaml`)

**Runtime** (containers):
- `jq` in route-init container (for JSON parsing)
- `wireguard-tools` in wireguard container
- `openvpn` in openvpn container

## Security Considerations

- `vpn-instances.yaml` contains credentials — must be
  gitignored like `.env`
- Credentials are passed as env vars to containers (same
  security model as current `.env` approach)
- WireGuard and OpenVPN config files may contain private keys —
  mount as read-only, gitignore

## Rollback Plan

Keep the old `docker-compose.yml` `l2tp-vpn` service definition
commented out. To rollback: delete override file, uncomment old
service, restore `.env` vars. Or simply `git revert`.
