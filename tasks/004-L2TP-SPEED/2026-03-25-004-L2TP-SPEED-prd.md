# 004-L2TP-SPEED: Company VPN Speed Investigation & Fix — PRD

**Status**: Draft
**Created**: 2026-03-25
**Author**: Claude (via dev workflow analysis)

---

## Context

Data transfer speed between local host and company network through the L2TP
VPN tunnel is significantly slower than general internet traffic routed through
Mullvad WireGuard. Company traffic achieves only 1–10 Mbps while Mullvad
internet traffic is substantially faster on the same VPS.

### Current State

The traffic path for company network access involves multiple encapsulation
layers:

```
Mac → Tailscale (MTU 1280) → gluetun (Docker bridge, MTU 1500)
    → L2TP container (Docker bridge) → ppp0 (MTU 1400) → company network
```

For comparison, Mullvad internet traffic:

```
Mac → Tailscale (MTU 1280) → gluetun → tun0 (WireGuard, MTU 1380) → internet
```

**Key infrastructure facts** (from Mar 23 investigation):

- **Latency is fine**: gluetun → company network shows 11ms, 0% loss
- **L2TP tunnel itself works**: MTU-sized (1400 byte) pings pass without
  fragmentation or loss
- **MSS clamping deployed**: TCPMSS rules clamp to 1240 bytes on tailscale0
  — this did NOT resolve the speed issue
- **Double NAT**: L2TP container masquerades on ppp0; gluetun masquerades
  on tun0 (but not on eth0 toward L2TP)
- **Routing**: company CIDRs routed from gluetun via Docker bridge to L2TP
  container, which forwards through ppp0
- **No MSS clamping on ppp0**: The L2TP container's mangle table is empty —
  MSS is only clamped on the Tailscale interface, not on the L2TP ppp0
  interface

### What MSS Clamping Fixed (and Didn't)

The existing TCPMSS rules on tailscale0 prevent fragmentation between Mac
and gluetun. However, TCP connections from gluetun → L2TP → company network
may still negotiate oversized segments because:

1. The Docker bridge between gluetun and L2TP has 1500 MTU
2. The L2TP ppp0 interface has 1400 MTU
3. No MSS clamping exists on ppp0 or on the Docker bridge path to L2TP

This means TCP SYN packets entering the L2TP container from gluetun may
carry MSS values based on 1500 MTU, causing fragmentation or drops at ppp0.

### Potential Root Causes to Investigate

1. **Missing MSS clamping on L2TP ppp0 interface** — TCP segments too large
   for the 1400-byte ppp0 tunnel
2. **L2TP encryption overhead** — IPsec ESP encapsulation adds ~60 bytes,
   further reducing effective payload
3. **xl2tpd/PPP performance** — userspace PPP processing may bottleneck
4. **IPsec/strongSwan crypto overhead** — encryption/decryption CPU cost
5. **Double NAT conntrack overhead** — connection tracking across two NAT
   layers
6. **TCP window scaling / buffer sizes** — suboptimal TCP parameters for
   high-latency VPN path
7. **Asymmetric routing or path issues** — return traffic taking a
   suboptimal path
8. **Company-side throttling** — the L2TP server or company firewall may
   rate-limit VPN connections

## Problem Statement

**Who**: Developer using the VPS as a Tailscale exit node for company access
**What**: Company network traffic is 5–50x slower than Mullvad internet traffic
**Why**: Slow file transfers, sluggish internal web apps, poor RDP/SSH
  responsiveness to company resources
**When**: All company network access through the L2TP tunnel

## Goals

### Primary Goals

1. **Tailscale must not depend on Mullvad.** Tailscale's control plane
   (HTTPS to Headscale), DERP relay, STUN, and WireGuard peer-to-peer
   traffic must route directly through the VPS host network, bypassing
   Mullvad completely. The current architecture routes all Tailscale
   traffic through Mullvad because Tailscale runs inside gluetun's
   network namespace. This makes the entire private infrastructure —
   Tailscale mesh, Headscale coordination, peer connectivity — dependent
   on a third-party VPN provider. If Mullvad goes down, all Tailscale
   connectivity dies: no auth, no DERP, no peer discovery, no access to
   any node. This is a critical architectural flaw that must be fixed.

2. **Company VPN throughput must match or exceed Mullvad speed.** The
   L2TP tunnel connects directly to the company network via IPsec, with
   fewer hops than Mullvad WireGuard. The current situation (company
   slower than Mullvad) indicates something is broken.

### Secondary Goals

- Document the network path performance characteristics for future debugging
- Add MSS clamping or MTU fixes to all relevant interfaces
- Establish a baseline speed measurement method for ongoing monitoring

## Requirements

### Functional Requirements

1. **FR-1**: Systematic per-hop speed measurement
   - **Priority**: High
   - **Rationale**: Must isolate which segment is slow before fixing
   - **Method**: iperf3 or similar between each container pair

2. **FR-2**: MSS clamping on L2TP ppp0 interface
   - **Priority**: High
   - **Rationale**: ppp0 has 1400 MTU but no MSS clamping — likely cause
     of fragmentation on company-bound TCP traffic
   - **Dependencies**: l2tp/entrypoint.sh modification

3. **FR-3**: TCP tuning for VPN path
   - **Priority**: Medium
   - **Rationale**: Default TCP parameters may not suit multi-hop VPN path
   - **Scope**: Buffer sizes, window scaling, congestion control algorithm

4. **FR-4**: Speed test automation
   - **Priority**: Low
   - **Rationale**: Ongoing monitoring and regression detection
   - **Scope**: Simple script or documented commands

### Non-Functional Requirements

1. **NFR-1**: No disruption to Mullvad internet traffic speed
2. **NFR-2**: Changes must survive container restarts
3. **NFR-3**: L2TP auto-reconnect behavior must not be affected

### Technical Constraints

- Must work within existing Docker Compose architecture
- Cannot modify company-side VPN server configuration
- Changes limited to: iptables rules, sysctl settings, PPP options,
  entrypoint scripts
- MSS/MTU changes must account for all encapsulation layers:
  Tailscale (1280) → Docker bridge (1500) → ppp0 (1400) → IPsec ESP (~60)

## Investigation Plan

### Phase 1: Baseline Measurements

Measure throughput at each segment to isolate the bottleneck:

| Segment | Method | Expected |
|---------|--------|----------|
| Mac → gluetun (Tailscale) | iperf3 | ~50+ Mbps |
| gluetun → internet (Mullvad) | curl speed test | ~50+ Mbps |
| gluetun → L2TP container (Docker bridge) | iperf3 | ~1 Gbps |
| L2TP container → company (ppp0) | iperf3 or curl to company resource | ??? |
| Mac → company (end-to-end) | curl/scp to company resource | 1-10 Mbps |

### Phase 2: Targeted Fixes

Based on Phase 1 findings, apply fixes in order:

1. Add MSS clamping on ppp0 in L2TP entrypoint
2. Tune TCP parameters (if buffer/window issue found)
3. Optimize IPsec settings (if crypto overhead is the bottleneck)
4. Consider alternative L2TP client (if xl2tpd is the bottleneck)

### Phase 3: Verification

Re-run Phase 1 measurements and compare.
